## 一、系统概述

### 1.1 这个平台是做什么的？

**网络工单平台**是一个"告警接收 → 工单生成 → 推送给客户 → 客户处理"的自动化系统。

**通俗理解**：
> 你的监控系统（如 Zabbix、Prometheus）发现服务器出问题时，会自动把这个"告警"发给本平台。平台把它变成一张"工单"，然后推送给对应的客户系统。客户收到后在自家系统里处理，处理完告诉平台"我搞定了"。

**举个例子**：

```
Zabbix 发现 "服务器 A CPU 使用率 95%"
        ↓
   推送给本平台
        ↓
   平台生成工单 #NT001
        ↓
   平台把工单推送给 "XX 公司运维系统"
        ↓
   XX 公司运维人员在自家系统里处理
        ↓
   XX 公司回调平台："已授权处理"
        ↓
   XX 公司处理完成后，工单状态变为"已完成"
```

### 1.2 平台里的三个"角色"

| 角色 | 说明 | 对应现实中的谁 |
|------|------|--------------|
| **管理员 (admin)** | 能操作所有功能，包括新增/删除客户 | 平台管理员、运维主管 |
| **操作员 (operator)** | 能查看工单、操作工单，但不能管理客户 | 值班工程师 |
| **客户系统** | 接收工单推送的外部系统 | 合作方的运维平台 |

### 1.3 系统架构简图

```
┌─────────┐    ┌──────────┐    ┌───────────┐    ┌──────────┐
│ 监控    │───▶│ 本平台   │───▶│  客户 A   │───▶│ 回调授权 │
│ 系统    │    │ (你正在  │    │  运维系统 │    │ 或拒绝   │
│ (Zabbix │    │  使用的) │    │           │    │          │
│ 等)     │    │          │    │  客户 B   │───▶│ 回调授权 │
└─────────┘    └──────────┘    │  运维系统 │    │ 或拒绝   │
                               └───────────┘    └──────────┘
```

### 1.4 登录信息

- **地址**：浏览器打开 `http://你的服务器地址/login`
- **默认账号**：`admin` / `admin123`
- **重要**：首次登录后请立即修改密码

## 二、快速开始

### 2.1 首次使用三步走

**第一步：创建告警源**

告警源 = "谁给平台发告警"。你需要告诉平台："我的 Zabbix 监控要接入"，平台就会给这个 Zabbix 分配一个专属的接收地址。

操作路径：告警源管理 → 新建告警源 → 填写名称和类型 → 保存

**第二步：创建客户**

客户 = "工单要推送给谁"。你需要告诉平台："收到告警后推给 XX 公司的运维系统"。

操作路径：客户管理 → 新建客户 → 填写名称、推送地址、API Key → 保存

**第三步：在监控系统里配置**

把平台给的 Webhook 地址填到你的 Zabbix / Prometheus 里。这样告警就会自动流进来了。

> 详细步骤见下文各模块说明。

### 2.2 管理员 vs 操作员权限对照

| 功能 | 管理员 | 操作员 |
|------|:------:|:------:|
| 查看工单列表 | ✅ | ✅ |
| 查看工单详情、重试、取消 | ✅ | ✅ |
| 查看告警源 | ✅ | ✅ |
| **新增/编辑/删除告警源** | ✅ | ❌ |
| 查看审计日志 | ✅ | ✅ |
| **客户管理（增删改）** | ✅ | ❌ |

## 三、功能模块详解

### 3.1 告警源管理

#### 3.1.1 什么是告警源？

**告警源**就是"谁给平台发告警"。常见类型：

| 类型 | 说明 | 适合谁 |
|------|------|--------|
| `zabbix` | Zabbix 监控系统 | 使用 Zabbix 的企业 |
| `prometheus` | Prometheus + Alertmanager | 云原生/容器化环境 |
| `generic` | 通用 JSON 格式 | 其他自定义监控系统 |

#### 3.1.2 创建告警源（操作步骤）

1. 进入「告警源管理」页面
2. 点击「新建告警源」
3. 填写以下信息：

| 字段 | 必填 | 填什么 |
|------|:----:|--------|
| 名称 | ✅ | 起个名字，如 "Zabbix 主监控" |
| 类型 | ✅ | 选择 `zabbix` / `prometheus` / `generic` |
| Webhook Secret | ❌ | 验证密钥，建议填一个随机字符串（16位以上） |
| 去重字段 | ❌ | 防止重复告警的字段，见下方说明 |
| 去重时间窗口 | ❌ | 默认 300 秒（5 分钟） |

4. 保存后，系统会分配一个 `source_id`（比如 `1`）
5. 你的专属 Webhook 地址就是：
   ```
   POST http://你的服务器地址/api/v1/alerts/webhook/1
   ```

#### 3.1.3 去重配置（重要）

**什么是去重？**

监控系统可能会对同一个问题重复发送告警。去重就是："同样的告警，不要给我重复创建工单，而是追加到已有的工单里"。

**怎么配置？**

在"去重字段"里填写 JSON 数组，比如 `["host.name", "event.name"]`。平台会把这两个字段的值拼起来算一个"指纹"，5 分钟内相同指纹的告警会被认为是同一个问题。

**示例**：

假设去重字段配置为 `["host", "alertname"]`：

| 时间 | 收到的告警 | 平台怎么处理 |
|------|-----------|-------------|
| 12:00 | `{"host":"srv01","alertname":"CPU高"}` | 创建新工单 #NT001 |
| 12:01 | `{"host":"srv01","alertname":"CPU高"}` | 追加到 #NT001（同一问题） |
| 12:02 | `{"host":"srv02","alertname":"CPU高"}` | 创建新工单 #NT002（不同服务器） |

> ⚠️ **已实现**：去重功能完全可用。

#### 3.1.4 在 Zabbix 里配置 Webhook

**步骤一：创建媒介类型**

1. 登录 Zabbix 管理端
2. 进入 Administration → Media types → Create media type
3. 类型选择 **Webhook**
4. URL 填入平台地址：`http://你的服务器/api/v1/alerts/webhook/你的source_id`
5. HTTP 方法选 `POST`
6. 如果配置了 Webhook Secret，添加 HTTP Header：`X-Webhook-Secret: 你的密钥`

**步骤二：配置 Action**

1. 进入 Configuration → Actions → Create action
2. 配置触发条件（比如 CPU > 90%）
3. Operations 中选择上面创建的 Webhook 媒介类型

> 平台会自动解析 Zabbix 发送的字段（主题、详情、主机名、IP、严重等级）。

#### 3.1.5 在 Prometheus 里配置

编辑 Alertmanager 的配置文件 `alertmanager.yml`：

```yaml
receivers:
  - name: 'network-ticket'
    webhook_configs:
      - url: 'http://你的服务器/api/v1/alerts/webhook/你的source_id'
        send_resolved: true   # 告警恢复也推送
```

#### 3.1.6 其他系统怎么接入（generic 类型）

如果你的监控系统不是 Zabbix 也不是 Prometheus，就用 `generic` 类型。只要你的系统能发送 JSON 格式的 HTTP POST 请求即可。

**平台能自动识别的字段**：

| 平台需要的字段 | 从 JSON 的哪里取 | 示例 JSON 字段名 |
|--------------|----------------|----------------|
| 标题 | `title` 或 `alertname` | `"title": "CPU 过高"` |
| 描述 | `description` 或 `message` | `"description": "server-01 CPU 超过 90%"` |
| 严重等级 | `severity` | `"severity": "critical"` |
| 源 IP | `source_ip` | `"source_ip": "192.168.1.100"` |
| 设备名称 | `device_name` | `"device_name": "server-01"` |

**最简单的接入方式**（扁平 JSON）：

```json
{
  "title": "CPU 使用率过高",
  "description": "server-01 CPU 使用率超过 90%",
  "severity": "critical",
  "source_ip": "192.168.1.100",
  "device_name": "server-01"
}
```

> ⚠️ **已实现**：generic 解析器完全可用，支持嵌套 JSON 和数组。

#### 3.1.7 轮询模式（配置界面有，但功能暂未实现）

在告警源配置页面，你可能会看到"轮询端点"和"轮询间隔"两个字段。

- **轮询模式**：平台主动去你的监控系统拉取告警（而不是等监控系统推送过来）
- **当前状态**：⚠️ 配置界面已显示这两个字段，但后端轮询调度器**尚未实现**。如需使用轮询，请联系开发团队。

### 3.2 客户管理

#### 3.2.1 什么是客户？

**客户**就是"工单要推送给谁"。一个客户对应一个外部系统。

> 只有管理员可以新增/编辑/删除客户。操作员只能查看。

#### 3.2.2 创建客户（操作步骤）

1. 进入「客户管理」页面
2. 点击「新建客户」
3. 填写以下信息：

| 字段 | 必填 | 填什么 |
|------|:----:|--------|
| 名称 | ✅ | 客户名称，如 "XX 公司运维系统" |
| 推送地址 | ✅ | 客户接收工单的 HTTP 地址，如 `https://ops.xxx.com/api/tickets` |
| API Key | ✅ | 客户身份标识，建议格式 `ak_xxx`，长度 16~64 位 |
| HMAC Secret | ✅ | 签名密钥，建议格式 `hs_xxx`，长度 32~128 位 |
| 回调地址 | ❌ | 可选，授权回调时平台会回传这个地址 |
| 状态 | ❌ | `active`（默认，正常推送）/ `inactive`（暂停推送） |

#### 3.2.3 API Key 和 HMAC Secret 是什么？

这两个字段是为了**安全**——确保推送的工单是真的来自本平台，而不是别人伪造的。

| 字段 | 用途 | 怎么给 |
|------|------|--------|
| **API Key** | 客户的"身份证号"，平台靠它识别"这是客户 A 还是客户 B" | 你自己编一个，告诉客户 |
| **HMAC Secret** | 签名密钥，平台推送时用它计算一个"签名"，客户收到后验证签名 | 你自己编一个，**必须保密**，只告诉客户 |

> ⚠️ 两端的 HMAC Secret 必须完全一致，否则签名验证会失败。

#### 3.2.4 客户的对接流程

客户（外部系统）需要做两件事：

**1. 接收工单推送**

平台会以 HTTP POST 方式推送工单到客户的"推送地址"，请求体如下：

```json
{
  "ticket_no": "NT20240101120000a1b2c3",
  "title": "CPU 使用率过高",
  "description": "server-01 CPU 使用率超过 90%",
  "severity": "critical",
  "alert_parsed": {
    "source_ip": "192.168.1.100",
    "device_name": "server-01"
  },
  "callback_url": "https://你的平台/api/v1/callback/authorization"
}
```

请求头里会带有签名信息，客户需要验证签名。

**2. 回调授权或拒绝**

客户处理完工单后，调用平台回调接口告知处理结果：

```json
{
  "ticket_no": "NT20240101120000a1b2c3",
  "action": "authorize",      // 授权处理，或 "reject" 拒绝
  "operator": "张三",
  "comment": "已确认，开始处理"
}
```

> ⚠️ **已实现**：推送协议和回调协议均完整实现。客户侧需要开发对接代码，平台侧无需额外开发。

### 3.3 工单管理

#### 3.3.1 工单是怎么产生的？

```
监控系统发送告警
      ↓
平台接收并解析
      ↓
检查是否重复（去重）
      ↓
创建新工单 或 追加到已有工单
      ↓
推送给关联的客户
      ↓
等待客户回调授权/拒绝
```

#### 3.3.2 工单状态说明

| 状态 | 颜色 | 含义 |
|------|------|------|
| **pending** | 🟡 黄色 | 新建，等待推送给客户 |
| **in_progress** | 🔵 蓝色 | 客户已授权，正在处理中 |
| **completed** | 🟢 绿色 | 处理完成 |
| **failed** | 🔴 红色 | 推送失败（重试耗尽） |
| **cancelled** | ⚪ 灰色 | 管理员手动取消 |
| **rejected** | 🟠 橙色 | 客户拒绝处理 |

#### 3.3.3 工单详情页能看到什么？

点击工单编号进入详情页，包含：

**基本信息**：
- 工单编号、标题、描述
- 严重程度（critical / warning / info）
- 告警来源（哪个 Zabbix/Prometheus）
- 关联客户
- 创建时间

**工作流时间线**（7 个节点的状态追踪）：

| 节点 | 含义 |
|------|------|
| alert_received | 告警已接收 |
| parsed | 告警内容已解析 |
| pushed | 已推送给客户 |
| awaiting_auth | 等待客户授权 |
| authorized | 客户已授权 |
| executing | 执行中 |
| completed | 已完成 |

每个节点的状态可能是：
- `done`（已完成）
- `active`（当前进行中）
- `failed`（失败）
- `skipped`（跳过，如客户拒绝）

**告警追加记录**：如果同一个问题在去重窗口内多次触发，会显示追加历史。

#### 3.3.4 手动操作

- **重试**：工单状态为 `failed` 时，点击「重试」重新推送
- **取消**：将工单状态置为 `cancelled`

> ⚠️ **已实现**：工单列表、详情、筛选、分页、重试、取消功能均可用。工作流时间线可视化已可用。

#### 3.3.5 工单超时（字段已预留，功能暂未实现）

数据库中有 `timeout_at` 字段，用于设置工单的处理截止时间。但当前版本：
- ⚠️ 没有超时扫描逻辑
- ⚠️ 没有超时通知功能
- ⚠️ 不会自动标记超时工单

如需超时提醒功能，请联系开发团队。

## 四、完整配置流程示例

### 场景：Zabbix 监控接入，推送给"XX 公司运维系统"

**第一步：创建告警源**

| 字段 | 填写值 |
|------|--------|
| 名称 | "Zabbix 生产监控" |
| 类型 | `zabbix` |
| Webhook Secret | `wh_zabbix_prod_2024`（随机字符串） |
| 去重字段 | `["host.name", "event.name"]` |
| 去重时间窗口 | 300 |

保存后得到 `source_id = 1`，Webhook 地址为：
```
POST http://你的服务器/api/v1/alerts/webhook/1
```

**第二步：创建客户**

| 字段 | 填写值 |
|------|--------|
| 名称 | "XX 公司运维系统" |
| 推送地址 | `https://ops.xxx.com/api/tickets` |
| API Key | `ak_xx_company_2024` |
| HMAC Secret | `hs_xx_company_secret_key_32chars` |

保存后得到 `client_id = 1`。

**第三步：在 Zabbix 里配置**

1. 创建 Media Type（Webhook）
2. URL 填 `http://你的服务器/api/v1/alerts/webhook/1`
3. HTTP Header 加 `X-Webhook-Secret: wh_zabbix_prod_2024`
4. 创建 Action，触发条件配好，Operations 选这个 Media Type

**第四步：通知 XX 公司**

把以下信息提供给 XX 公司的技术人员：

| 信息 | 值 |
|------|-----|
| API Key | `ak_xx_company_2024` |
| HMAC Secret | `hs_xx_company_secret_key_32chars` |
| 回调地址 | `http://你的服务器/api/v1/callback/authorization` |

XX 公司需要在自家系统里：
1. 开发接收工单的接口（验证签名后解析工单数据）
2. 开发回调授权的接口（处理完工单后调用平台回调接口）

**第五步：测试**

在 Zabbix 里手动触发一个测试告警，观察：
1. 平台是否收到告警（工单列表是否有新工单）
2. 工单是否推送给 XX 公司
3. XX 公司回调后工单状态是否变为 `in_progress`

## 六、常见问题

### Q1: 告警推送到平台后，工单没有自动推送给客户？

**A**: 当前版本存在一个已知问题：工单创建后，自动推送任务的触发逻辑尚未连接。这意味着：
- 平台会正常接收告警并创建工单 ✅
- 但工单不会自动推送给客户 ❌

**临时解决方案**：联系开发团队修复，或手动通过 API 触发推送。

### Q2: 怎么修改管理员密码？

**A**: 当前版本没有前端密码修改页面。需要：
1. 直接修改数据库 `users` 表的 `password_hash` 字段（bcrypt 加密后的密码）
2. 或联系开发团队添加密码修改功能

### Q3: 可以新增操作员账号吗？

**A**: 当前版本没有前端用户管理页面。需要直接操作数据库：
```sql
INSERT INTO users (username, password_hash, role) 
VALUES ('operator1', '<bcrypt_hash>', 'operator');
```
或使用后端 API（需 admin JWT Token）。

### Q4: 平台支持短信/邮件/钉钉通知吗？

**A**: 当前版本不支持。通知渠道（短信、邮件、钉钉、企业微信等）属于规划中的功能，v1.5 及之后版本可能加入。

### Q5: 可以一个告警推送给多个客户吗？

**A**: 当前版本一个工单只关联一个客户。如需多客户推送，需要：
1. 创建多个客户
2. 为每个客户分别配置推送
3. 或联系开发团队评估多客户推送功能的开发

### Q6: 部署后访问不了？

**A**: 按以下步骤排查：

1. 检查服务状态：`./manage.sh status`
2. 查看日志：`./manage.sh logs`
3. 确认 MySQL 已启动：`./manage.sh logs mysql`
4. 测试后端直连：`curl http://localhost:8080/api/v1/auth/login`
5. 检查 Nginx 配置是否正确指向后端服务
